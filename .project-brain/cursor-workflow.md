# Cursor 工作规范

> 本文档约束 Cursor（或任何 AI 协作工具）在 BinxinAdmin 项目里的工作方式。
> 由真实事件沉淀而来，违反此规范的产出会被 review 拒绝。

---

## 一、修复 bug 前必须过的"四步自检清单"

来源：2026-05 task-005 字典接口 mock 兜底事件的反思。

### Step 1：先回到任务书 / 设计意图原文

确认"**应该是什么样**"。

不要凭印象修，不要凭"通常这样做"修。**先找原始约定**——任务书、ADR、过往决策日志里的相关条款。

### Step 2：对照"现在的实现是什么样"

找出**实现与设计之间的差距**。

差距才是真正的 bug 根因。表层症状（比如 Console 报错、404）只是差距的外在表现。

### Step 3：修复描述里先讲设计差距，再讲技术手段

**禁用"听起来合理"的技术名词包装绕过**。

错误示范：
> "Network 的 404 是浏览器原生行为，JS 拦截不掉，所以改了实现。"
> （把锅推给浏览器，掩盖原始设计偏差）

正确示范：
> "原实现违背任务书'mock 优先'的意图——既然约定 mock 优先，就根本不该发请求。
> 修复方式：已 mock 的字典直接 return。"
> （先讲设计差距，再讲技术手段）

### Step 4：用户报某个表层症状时，警惕"只修症状不修因"的诱惑

用户看到的是症状（404 / 报错 / 白屏 / 错位）。

修症状的代价小，修因的代价大。但**只修症状会埋雷**——同样的因可能在别处再爆。

每次修 bug 都问自己一遍："这是症状还是因？"

## 二、其他工作规范

### 2.1 任务书与共享包 API 不一致时
不要硬执行任务书。**主动列出对齐项**，给出处理方案，等 review 通过再实施。
（来源：task-005 实施时 Cursor 主动列了 7 个对齐项的优秀做法）

### 2.2 commit 规范
- 必须用中文（D-028）
- 前缀保留 feat/fix/docs 等约定式英文关键字
- 冒号后用中文描述
- 示例：`feat: 实现JWT三守卫与跨类型Token拒绝`

**前缀语义必须符合仓库实际状态：**
- 仓库**首次 commit**（No commits yet）→ 必须用 `feat:` 描述基线建立，不能用 `fix:`/`docs:`
- 已有基线后的修复 → `fix:`
- 已有基线后的新特性 → `feat:`
- 已有基线后的文档 → `docs:`

**反例**（曾在 task-005 admin-web 首次 commit 时差点出现）：仓库 No commits yet 时
用 `fix: 修复XXX` 当首次 commit。这会让 git 历史出现"先 fix 后 feat"的诡异语义，
未来翻历史的人会困惑"被 fix 的 feat 在哪里"。**审 commit message 前必须 git status / git log 看仓库状态。**

### 2.3 凭据安全
- 严禁明文密码/密钥/Token 进入任何 git 跟踪的文件（含 .project-brain）
- 输出敏感信息默认写本地 gitignore 文件，CLI 仅提示路径
- 提交文档前 grep 扫一遍敏感字符串
（来源：D-030 / D-033，两次密码事故）

### 2.4 共享包变量契约原则
**完整 > 稳定**。发现合理遗漏的变量应立即补，而非在调用方用 :deep 私下解决。
（来源：D-037，task-005 实施时 sidebar 变量补全事件）

### 2.5 修复前先讲计划
任何超过 5 个文件的变更，先列执行计划等用户确认，再开始改。
不要"边改边问"——计划阶段问完所有问题，实施阶段专注执行。

### 2.6 验证不能只靠编译时检查
- type-check / lint / build 通过 ≠ 功能正常
- 用户场景必须由浏览器/真实环境验证
- 任务书要求的"X 项验证清单"必须逐项跑通，不能跳过
（来源：task-005 完成后 Pinia 双实例 bug 是由浏览器才发现的）

## 三、当 Cursor 不确定时

不确定优先级：
1. **回查项目大脑**（00-CHARTER / 01-ARCHITECTURE / 03-CONVENTIONS / 05-DECISIONS-LOG / 相关任务书）
2. **问用户**（用 ask_user_input 或对话提问），不要擅自决定
3. **给出 2-3 个可选方案 + 推荐**，让用户拍板

禁用："我猜你想要"、"通常情况"、"应该是这样"作为决策依据。

## 四、凭据规范的最终形态（从三次密码事故得出）

任何形态的密码/密钥/Token 都**绝不**出现在以下位置：
- 任何 git 跟踪的文件（代码、文档、.project-brain、Changelog 等）
- 任何 CLI stdout 输出（seeder/命令行工具）
- **任何交付报告/对话/聊天/截图/邮件**

正确做法：
1. seeder/CLI 把密码写入项目根/database 等位置的 `.local-credentials.json`（.gitignore 必须屏蔽）
2. 文件权限 0600
3. stdout 只输出引导：`密码已写入 {完整路径}，请立即妥善保存或登录后修改`
4. 交付报告凭据部分**标准格式**：
   ```
   演示账号位置: database/.local-credentials.json（gitignore，不入仓库）
   用户名: <username>
   租户/作用域: <scope>
   密码: 见上述文件
   ```
5. 用户自己进项目目录 `cat .local-credentials.json` 一次，把密码抄到密码管理器，然后 `rm` 或保留（视情况）

**反例**（任何一条都视为事故）：
- `echo $password` 在 seeder 输出
- 交付报告写"密码: xxx"或在表格里贴出密码
- 在聊天/邮件里告诉对方密码

合并 git 之前 grep 扫描凭据残留：
```bash
grep -rn "<密码字符串前缀>" <project-root>
# 应该只返回 .local-credentials.json，其他位置 0 行
```
